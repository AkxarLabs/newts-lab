## Detail — critical & high

### [CRITICAL] compete — `templates/compete/scripts/report_score.py:182`
- **What:** TOCTOU race on the unlocked append-only scores.jsonl lets concurrent sessions exceed the PI-signed external-read cap (the Gate-2 analogue is not actually enforced under concurrency).
- **Trigger:** score_envelope is total_max:5, pi_signed:true, 4 reads already logged. The protocol explicitly supports concurrent project sessions (agent_runner.py) and unattended /research-loop. Two report_score.py invocations start near-simultaneously; both call _ledger_rows() at line 182 and each see len(reads)==4 < 5, so both pass the cap check, both run the external score_command (two real leaderboard submissions), then both _append() a read row at line 223 -> 6 logged reads against a 5-cap envelope. The over-limit outward actions already happened. No file lock exists anywhere in the read-check/append path, so the cap the spec says is 'enforced HERE' is silently breachable.
- **Fix:** Make the cap-check-and-append atomic under an OS file lock held across lines 182-223: acquire an exclusive lock on scores.jsonl (msvcrt.locking on Windows / fcntl.flock on POSIX, or a sidecar .lock file created with O_CREAT|O_EXCL and retried), re-read the ledger inside the lock, re-verify total_max/per_day_max, run the command, append the read row, then release. Document that report_score.py serializes external reads.

### [CRITICAL] s2 — `tools/s2.py:213`
- **What:** verify marks a real, correctly-cited paper as NOT-FOUND (exit 1, the 'fabricated citation' signal) whenever S2 is merely unreachable or rate-limited — the exact 'empty result is NOT evidence of absence' failure the spec forbids, but with no exit-3 escape hatch in verify.
- **Trigger:** Run `verify references.bib` while S2 is down / returns 500 / exhausts its 429 retries. http_get returns None (line 211 match=None), so `found = (None or {}).get('data') or (...)` evaluates to [] (line 213), the code prints `NOT-FOUND | no S2 match for title` and does `worst = max(worst, 2)` (line 216). A bibliography of 100 genuine, perfectly-cited papers all report NOT-FOUND and cmd_verify returns exit 1, which /review-paper interprets as fabricated/missing citations — blocking a paper on a transient network blip and asserting evidence of absence the search-side code explicitly refuses to assert.
- **Fix:** Distinguish 'network/backend failure' from 'genuine no-match'. Have http_get signal unreachable distinctly (e.g. return a sentinel or raise on non-404 failures vs return None only on 404/empty), then in cmd_verify when the match request failed for non-404 reasons, emit a BLOCKED/UNVERIFIED status and propagate an exit-3-style 'verification blocked' code instead of NOT-FOUND, mirroring the search command's both-unreachable contract.

### [HIGH] agent-runner — `tools/agent_runner.py:196`
- **What:** _guard_extra uses exact-token equality (`f in extra.split()`), so any `=`-joined form of a forbidden flag bypasses the human-in-loop sandbox/permission guard.
- **Trigger:** PI sets `agents.programmatic.backends.codex.extra_args: "--sandbox=danger-full-access"` (or claude `--permission-mode=bypassPermissions`). `extra.split()` yields the single token `--sandbox=danger-full-access`, which is != the forbidden token `--sandbox`, so `hit` is empty, no SystemExit is raised, and the token is appended to argv at line 218. codex's clap parser accepts the `=` form, so the launched agent runs with full filesystem access and no approvals — exactly the human-in-loop default the guard promises to protect. The tool silently grants what it claims to refuse.
- **Fix:** Normalize each token before comparison: split on `=` and compare the flag head, e.g. `hit = [f for f in forbidden if any(tok == f or tok.startswith(f + '=') for tok in extra.split())]`. Add a regression test passing the `=` form to assert SystemExit is still raised.

### [HIGH] audit — `tools/audit_claims.py:216`
- **What:** A claim with an empty (or omitted) `numbers` list is reported PASS, verifying nothing — a silent hard-rule-1 traceability bypass.
- **Trigger:** A claims.yaml entry like `{id: C001, project: demo, artifacts: [runs/r0/metrics.json]}` with no `numbers:` (or `numbers: []`, e.g. an author forgot to fill it). At line 171 `numbers = claim.get('numbers') or []` becomes `[]`; the `for n in numbers:` loop at line 216 never runs; `misses` stays empty and `all_direct` stays True; the function returns ('PASS','all numbers found directly') at line 232-233. The claim exists in the ledger and is mapped to an artifact but ZERO numbers were checked, yet the row prints PASS and contributes nothing to the FAIL count, so the blocking audit exits 0 and the paper passes review with an unverified claim.
- **Fix:** After computing `numbers`, treat an empty list as a defect: `if not numbers: return 'FAIL', 'claim has no numbers to verify (hard rule 1: a claim that checks nothing must not pass)'` (or 'MANUAL' only if a `derivation:` is present). This makes an empty-numbers claim block exactly like an unmatched one.

### [HIGH] audit — `tools/audit_claims.py:253`
- **What:** A paper whose claims.yaml has zero claims prints a 'suspicious' warning but still exits 0 (PASS), so the blocking audit does not block a paper that simply declares no claims.
- **Trigger:** studies/<slug>/paper/claims.yaml is the unedited template (`claims: []`) or any file with an empty claims list. Line 253 yields `claims = []`; line 254-255 prints the 'nothing to audit … suspicious' message but does NOT return. The loop body is skipped, `counts` stays all-zero, and unless --no-coverage is set the only backstop is coverage_scan — which only fires on DECIMAL/percentage numerals in main.tex. A results paper that reports only integer counts (e.g. '47 wins, 12 losses', no decimals) has an empty coverage list too, so line 281 `if counts['FAIL'] or coverage` is False and main() returns 0. The reviewer ensemble sees a green claims audit on a paper with no traced claims.
- **Fix:** Make zero-claims blocking: after the warning print, `return 1` (a results paper with no claims is a FAIL by hard rule 1), or at minimum gate exit on it — `if not claims and not args.no_coverage: ...` is insufficient because integer-only papers escape coverage. Safest: zero claims → exit 1 with the explicit message.

### [HIGH] bus — `tools/lab_bus.py:99`
- **What:** A directive's resolved state is computed from the LAST directive_* ack only, so a later non-terminal `seen` ack overrides a prior terminal `done`/`blocked` and resurrects a completed directive into the inbox — risking double execution of a command.
- **Trigger:** PI issues command directive d-001 = {action:"kill"}. Agent executes it and acks: `lab_bus.py ack d-001 done`. At a later orientation checkpoint the agent (or a re-run) emits `directive_seen` for the directives it reads before filtering, producing a `directive_seen` event for d-001 AFTER the `directive_done`. In `unresolved_directives`, `acked["d-001"]` is overwritten in file order to `directive_seen`; the terminal-check `if last in ("directive_done","directive_blocked")` is now False, so d-001 reappears as pending and the kill is executed a second time.
- **Fix:** Make terminal acks sticky: track whether ANY terminal ack exists for a ref, not just the last one — e.g. `terminal=set(); for e in events: if kind in (directive_done,directive_blocked): terminal.add(ref)` and skip any d['id'] in terminal regardless of later events.

### [HIGH] checks — `templates/project/scripts/reconcile.py:73`
- **What:** A healthy long-running run that has not yet (or only sparsely) written metrics is falsely flagged 'dead', because the staleness anchor falls back to the never-updated meta.json mtime.
- **Trigger:** A FULL run starts, writes meta.json (status=running) at T0, and is a slow-logging job that emits its first metrics line only every 10 min while log_interval_seconds=60. metrics.jsonl is empty (or absent) for the first minutes, so `anchor = stream if (stream.exists() and stream.stat().st_size) else meta` selects meta.json. meta.json is written once at start and never touched, so age = now - meta.mtime keeps growing; after 2*60s the live, healthy run is reported as 'Dead/stalled — the run process is gone', telling the operator to finalize its ledger entry or delete the dir mid-run. This directly contradicts the project CLAUDE.md note that a healthy sparse-logging run must not be flagged stalled.
- **Fix:** Only treat a running job as dead when it has actually produced a metrics stream that then went stale: skip the dead check entirely when metrics.jsonl is empty/absent (anchor on meta only to detect orphans, not deadness), or check the most-recently-modified file in the run dir (e.g. max mtime over d.glob('*')) rather than meta.json, and/or honor a configurable grace period before the first metrics write.
